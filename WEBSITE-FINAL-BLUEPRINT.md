# Website Final Blueprint

This document defines the stricter final direction for the portfolio.

It answers:

- exact page names
- exact navigation labels
- exact homepage section order
- sci-fi vs gothic ratio
- what stays on the homepage
- what moves to subpages

The goal is to turn the current portfolio into a clear, themed, and maintainable final structure.

---

## 1. Final Site Direction

### Core identity

The portfolio should feel like a:

- `character summary screen`
- `developer dossier`
- `mission codex`

The site should present Tanvir as:

- a `Gameplay Programmer`
- with strong `systems`, `tools`, and `technical writing` depth
- using a game-theme interface without sacrificing clarity

### Final design rule

`Readable first. Themed second. Memorable throughout.`

That means:

- every themed element must still support scanning
- style should reinforce meaning
- nothing important should be buried behind gimmicks

---

## 2. Final Visual Ratio

### Recommended ratio

- `70% sci-fi`
- `30% gothic`

### What that means in practice

#### Sci-fi should define:

- layout system
- panels
- stat blocks
- mission cards
- navigation language
- UI structure
- overlays
- interactions

#### Gothic should influence:

- mood
- lighting
- texture
- accent atmosphere
- subtle ornament
- emotional tone

### What to avoid

Avoid going:

- full cyberpunk neon
- full fantasy parchment
- full horror/grunge

The result should feel like:

`a dark sci-fi operations interface with gothic atmosphere`

not:

`a haunted game menu`

---

## 3. Final Navigation Labels

### Top navigation

Use these exact labels:

1. `Home`
2. `Game Dev`
3. `Tools`
4. `Writing`
5. `Contact`
6. `Resume`

### Why these labels

- `Home` keeps orientation simple
- `Game Dev` is the clearest public-facing label for Unity/gameplay work
- `Tools` is cleaner than `Tools / Web` in navigation
- `Writing` is shorter and stronger than `Writing / Publishing`
- `Contact` is obvious
- `Resume` is a normal professional expectation

### Resume behavior

For now:

- `Resume` should open `Request Resume`

Later:

- it can link directly to a public PDF if you want one live

## 5. Homepage Final Section Order

The homepage should be a top-down `character page summary`.

Use this exact section order:

### 1. Character Summary

Purpose:

- establish identity immediately

Include:

- Name
- Class / role
- specialization line
- short positioning statement
- contact CTA
- GitHub / LinkedIn / Resume quick actions

### 2. Live Showcase

Purpose:

- give immediate interactive proof

Include:

- playable micro showcase
- short instructions
- tiny status readout

### 3. Core Stats

Purpose:

- communicate strongest abilities quickly

Include:

- gameplay systems
- Unity
- C#
- tooling
- combat readability
- AI / behaviors
- architecture
- technical writing

### 4. Skill Tree

Purpose:

- organize skills by branch
- connect skills to projects

Include branches:

- Gameplay
- Tools
- Web / .NET
- Writing

### 5. Mission Log

Purpose:

- show active projects as mission cards

Include:

- Unity RPG Prototype
- Game Bug Tracker
- GenericRPG in C#

Each card should support:

- status
- focus tags
- last updated
- short summary
- inline dossier toggle
- link to full page

### 6. Achievements / Unlocks

Purpose:

- show momentum and proof of long-term work

Include examples:

- 200 chapters written
- 8-volume series in progress
- custom domain live
- professional email configured
- interactive portfolio launched

### 7. Build Journey

Purpose:

- show growth over time

Should summarize:

- C# start
- console RPG systems
- tooling / architecture
- Unity focus
- current portfolio direction

### 8. Toolkit / Inventory

Purpose:

- show tools in a compact, visually satisfying block

### 9. Contact / Exit Paths

Purpose:

- close with clear next actions

Include:

- email
- GitHub
- LinkedIn
- Resume request

## 6. What Stays on Homepage

The homepage should keep:

- identity
- strongest stats
- top skills
- project summary cards
- highlights of achievements
- short build journey
- contact actions

The homepage should **not** become:

- a full archive of every detail
- a place for long technical writeups
- a cluttered gallery

### Homepage rule

If a section needs more than a `clear summary`, it probably belongs on a subpage.

## 9. Homepage vs Subpage Split

### Homepage = summary

Think:

- `Who am I?`
- `What do I build?`
- `What am I strongest at?`
- `What should you click next?`

### Subpages = category overview

Think:

- `What kind of work exists in this branch?`
- `How does this branch connect to the larger portfolio identity?`

### Project pages = proof

Think:

- `Here is the detailed evidence.`

---

## 10. Final Tone Rules

The site voice should be:

- confident
- precise
- atmospheric
- professional
- reflective

The site voice should avoid:

- exaggerated claims
- fake “epic” language
- edgy roleplay writing
- too much jargon without payoff

### Best voice style

`A serious developer speaking through a game-like interface.`

Not:

`a character pretending to be a developer`
